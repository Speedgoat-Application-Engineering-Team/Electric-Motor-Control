Description

Goal of this exercise is to test functioning of entire motor control software, including supervisory logic, mode scheduler, fault detection logic and input sensor processing logic. By bringing all the software components into one test harness (Simulink Model), it allows us to test the interaction between various components and find integration issues and design errors early in design phase. For this exercise, we will test our software against below requirements from Functional Requirement Document.
  1. Controller shall remain in normal operating mode (Closed-Loop Velocity Control) if requested command type is Velocity Control and all faults are absent
  2. Controller shall disable Inverter upon detecting Phase Currents greater than 30A within 10miliseconds
  3. Controller shall calibrate Encoder Offset before entering normal operating mode (Closed-Loop Velocity Control)
  4. Controller shall remain in Encoder Calibration mode when user requests Calibrate Encoder mode
Before we test our software for functional requirements, we need to ensure proper integration of all the software components. To test integration of different software modules, we will use FieldOrientedController_MIL.slx. Example of simple integration issues are:
  1. Data type mismatch between inputs and outputs of two separate modules
  2. Missing Port Connection
  3. Wrong connection of ports etc
These issues can be identified and resolved by successfully simulating all the software components into one model. However, we still need to test the functionalities to ensure software is working as per the requirements.
To test functional requirements, we will use desktop simulation to gain reasonable confidence in our design. Here we are going to use ad hoc/interactive testing method using Dashboard (User Interface) blocks to test functionalities. For systematic testing, MathWorks offers Simulink Test which helps engineers author test cases & test assessments; create and manage test plans; and automate testing, including real-time testing using Simulink Real-Time and Speedgoat Target Hardware.
After gaining high confidence in our design, we can test it on our hardware prototype of final product. Here, we will use Speedgoat Target Machine and Brushless DC Motor + Inverter Demo Kit.

Interactive Model-In-the-Loop Testing using Desktop Simulation Test Bench

Open Desktop Simulation Test Bench

mdl = 'FieldOrientedController_MIL';
open_system(mdl);
switchConfig('MIL');

Select Manual Test Input Options (Dashboard)

set_param([mdl,'/enManualTestInputs'],'Value','1');

Run Simulation and View Results

open_system('FieldOrientedController_MIL/User_Interface');
%set_param(mdl,'SimulationCommand','Start');
Simulink.sdi.view;
Using Dashboard blocks from Simulink, we can test our models interactively while simulation is in progress. In the figure below, we can see the results from an example run, where at ~1sec, "commandType" Input was changed from Calibrate_Encoder to Closed_Loop, which resulted into "mode" Output transitioning from Calibrate_Encoder to Closed-Loop. And rotor Velocity changed from Zero to ~200rad/sec following the target speed. At ~7.5 Second, Motor Off command was issued, which resulted into rotorVelocity slowly dropping to zero.
